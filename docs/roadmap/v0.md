# v0

This is the development release series of `betto_onnxrt`. The goals and their
current completion are as follows:

| # | Goal | % Complete |
|---|------|-----------|
| 1 | Deliver stable ORT support across all target platforms — pure Dart (CLI and server) and Flutter (iOS, Android, macOS, Windows, Linux). Web is excluded: the browser provides no FFI mechanism and no `dart:io`. | 95% |
| 2 | Allow integrators to configure allowlists for models that can be downloaded at runtime, with the option to permit any model. | 100% |
| 3 | Support the full range of ONNX output element types (not just `float32`) so the inference API is useful with any model architecture. | 100% |
| 4 | Ensure testing pipelines cover all supported platforms with real ORT load and inference, so that a regression on any platform cannot reach users behind a green CI badge. | 100% |
| 5 | Provide complete API documentation suitable for pub.dev publication. | 100% |
| 6 | Define a repeatable, documented process for adopting new ORT versions. | 0% |
| 7 | Build out a sample command-line tool based on Google's Magika model. | 5% |
| 8 | Publish `betto_onnxrt` and `betto_onnxrt_ios` to pub.dev in lockstep. | 0% |

The `betto_onnxrt` must be pure Dart - it must not require use of the Flutter
SDK.

### [x] BLOCKER: Wire the CI gate to real ORT inference

`make test` / `make cicd` skips every `onnx_session_test.dart` case when the ORT
binary is not staged, leaving `runtime.dart`, `session.dart`, and `ort_api.dart`
at 0% coverage in CI. A change that breaks FFI loading passes the gate cleanly.

Resolved:

- **Linux CI** (`build` job): `dart pub get` downloads `libonnxruntime.so.{ver}`
  to `.dart_tool/betto_onnxrt/{ver}/`; a `ln -sf` step creates the unversioned
  symlink and exports `LD_LIBRARY_PATH`, so `dart test` (JIT mode) can open the
  library by name. `onnx_session_test` now runs (not skips) on Linux CI.
- **Windows CI** (`test-windows` job): `onnxruntime.dll` is already unversioned;
  the hook cache directory is added to `PATH` so `dart test` finds it.
- **macOS CI** (`test-macos` job): JIT mode cannot open the macOS framework path
  that only exists after a Flutter/AOT build, so dart unit tests still skip on
  macOS. `make macos_test` (Flutter integration test, AOT) now runs in CI and is
  the authoritative ORT load+inference check for macOS.
- **Fail-not-skip on CI**: `onnx_session_test.dart` `setUpAll` fails immediately
  (rather than skipping) if the binary is absent on Linux or Windows CI, so a
  misconfigured pipeline is immediately visible.
- Makefile: `prepare_dart` / `prepare_flutter` / `clean_dart` split so Linux and
  Windows CI targets don't require Flutter SDK; `cicd_macos` now includes
  `macos_test`.

### [x] BLOCKER: Fix iOS on-device failure

`OnnxRuntime.load()` crashed on-device with `symbol not found: OrtGetApiBase`
because `betto_onnxrt_ios` was not wired into `integration_test_app`.

Resolved in `docs/plans/completed/plan_ios_integration_test.md`:

- Added `betto_onnxrt_ios` as a path dependency of `integration_test_app` and
  regenerated `GeneratedPluginRegistrant.m` — `BettoOnnxrtIosPlugin` is now
  registered.
- `make ios_test` + `nm -gU` symbol verification must be run manually by the
  developer (requires Xcode + SPM network fetch; cannot run in CI agent).
- If `_OrtGetApiBase` is absent after build, apply Option A (`-u _OrtGetApiBase`
  in xcconfig) or a Swift re-export bridge for the hidden-visibility case.

### [x] BLOCKER: Restore SHA-256 verification on desktop and iOS

Resolved in `docs/plans/completed/plan_sha256_desktop.md`.

- Introduced `version_onnx.json` at the repo root as the single source of
  truth for all platform binary metadata (versions, URLs, SHA-256 digests).
- Replaced all placeholder zeros with real SHA-256 digests for ORT v1.22.0
  (macOS arm64, Linux aarch64/x64) and v1.22.1 (Windows arm64/x64).
- Removed all all-zeros bypass clauses from `_isValid()` and `_ensureFile()`.
- Desktop now uses archive-level verification with a `.sha256` sidecar for
  the cache fast-path; Android retains two-level (archive + per-ABI) verification.
- iOS SHA-256 (SPM 1.24.2) recorded in `version_onnx.json` for reference;
  not read at build time (hook exits before manifest lookup on iOS).
- Added `test/version_manifest_test.dart` asserting no all-zeros values and
  anchored `^[0-9a-f]{64}$` format across all digest fields.

### [x] Linux and Windows integration tests

These platforms have never had inference exercised under an automated test —
they are trusted purely by analogy to macOS.

**Linux** — pure Dart apps can be tested on Linux; Flutter Desktop on Linux
cannot. Define a `make linux_test` target that runs the Dart integration tests
against the native ORT binary on a Linux runner. Flutter Desktop coverage on
Linux remains out of scope for v0.

**Windows** — pure Dart apps can be tested via the existing GitHub Actions
Windows runner. Define a `make windows_test` target and wire it into CI so
Windows Dart inference runs in the pipeline. Flutter Desktop coverage on Windows
remains out of scope for v0.

Until these targets are wired in, document a written manual smoke checklist for
Flutter Desktop on both platforms.

Resolved in `docs/plans/completed/plan_testing_pipeline.md`:

- Added `make linux_test` target to `Makefile` with `v`-prefix strip matching
  `cicd_linux` (reads `VERSION_ONNX`, strips leading `v`, sets `LD_LIBRARY_PATH`).
- Added `make windows_test` as a local-convenience target (no CI step needed;
  `cicd_windows` already runs real inference via `dart test`).
- Created `docs/manual_checks.md` covering Flutter Desktop smoke testing on
  Linux and Windows: Flutter channel/version, load-verification, failure
  signatures, and where to record evidence in the PR description.
- Rewrote `docs/spec/README.md` "Windows and Linux integration testing" section
  to reflect the new targets and the already-wired CI inference.

### [x] ORT vtable-slot guard test

Slot indices in `lib/src/ort_api.dart` are hand-maintained against
`onnxruntime_c_api.h` with no automated guard. A slot drift or typo calls the
wrong C function and may produce silent wrong output rather than a crash.

- Add a guard test that validates slot indices against the ORT header (or a
  recorded golden).
- Fix existing comment drift in `session.dart` (references "slots 31/32/33"
  where the code correctly uses 65/61/62).
- Enforce that any PR editing slot numbers or bumping `VERSION_ONNX` requires a
  real load+inference run, not just `dart test`.

Resolved in `docs/plans/completed/plan_testing_pipeline.md`:

- Added `// SLOT:Name=N` annotations on all 22 bound typedef pairs in
  `lib/src/ort_api.dart` (e.g. `// SLOT:CreateEnv=3`).
- Created `test/ort_slot_guard_test.dart` with a golden table for ORT API v22;
  the test reads `ort_api.dart` as text, extracts all `SLOT:(\w+)=(\d+)` pairs,
  and asserts they match the golden.
- Fixed comment drift in `session.dart`: method docstring and three inline
  comments changed from `(slot 31/32/33)` to `(slot 65/61/62)`.
- Added PR-enforcement guidance to `ort_api.dart` file-level docstring and
  `CLAUDE.md`: any PR editing slot indices or bumping `ortApiVersion` must
  include evidence of a passing `make macos_test` or `make linux_test` run.

### [x] Isolate thread-affinity test

`OnnxSession` is documented as thread-affine but nothing verifies the contract
or the cross-isolate failure mode.

- Add an integration test that asserts correct behaviour when a session is used
  from the creating isolate and documents (or tests) the failure mode when
  `run()` is called from a different isolate.

Resolved in `docs/plans/completed/plan_testing_pipeline.md`:

- Added `group('OnnxSession — thread affinity', ...)` to
  `test/onnx_session_test.dart`: calls `session.run()` three times sequentially
  from the same isolate and asserts all outputs equal `[1.0, 2.0, 3.0, 4.0]`.
- Added `// KNOWN: cross-isolate use is UB — not automated` comment documenting
  why the negative path is not automated (ORT mutex corruption is unsafe in CI).
- Strengthened `OnnxSession` class docstring to name the failure mode explicitly:
  "Calling `run()` or `dispose()` from a different isolate can corrupt ORT's
  internal thread-pool mutex state and produce undefined behaviour (crash or
  silent wrong output)."

### [x] Tighten SPM version pin

The `from: "1.22.0"` SPM pin in `betto_onnxrt_ios` was a minimum-version
constraint that could silently pull a newer ORT version and shift vtable slots.

Resolved in `docs/plans/completed/plan_ios_integration_test.md`:

- Changed `packages/betto_onnxrt_ios/ios/Package.swift` from `from: "1.22.0"`
  to `exact: "1.24.2"` (the SPM repo has no tags between 1.20.0 and 1.24.1;
  1.24.2 is what `from: "1.22.0"` already resolved to).
- Updated the `check_ios_version` Makefile target to grep for `exact:` and
  compare against `version_onnx.json ios.version` rather than `VERSION_ONNX`,
  since the two legitimately differ. `make check_ios_version` passes.

## [x] Output element-type introspection

`OnnxSession.run` now reads the output element type from each native `OrtValue`
via `GetTensorElementType` (vtable slot 60) and returns the appropriate
`TypedData` subtype. All five supported element types are handled: `float32`,
`uint8`, `int32`, `int64`, and `float64`.

Resolved in `docs/plans/completed/plan_output_element_types.md`.

Integration tests use BGE Small En v1.5 (small, English-only) and BGE-M3
(multilingual, larger) as the primary float32 models — both are embedding
models and work today without element-type introspection.

## [ ] API documentation completeness

pub.dev scores packages on documentation coverage, and a poor score affects
discoverability and trust. The `dart doc` target already exists in the Makefile;
this item ensures the docs are publication-ready before the pub.dev delivery
item is reached.

- Run `make doc` and review the generated output for missing or placeholder doc
  comments across all public API types (`OnnxRuntime`, `OnnxSession`,
  `OnnxTensor`, `OnnxElementType`, `SessionOptions`, `ModelDownloader`,
  `ModelSpec`, `ModelFile`, `ResolvedModel`, `AllowlistProvider`).
- Verify that the package-level library doc in `lib/betto_onnxrt.dart` gives a
  clear overview suitable for the pub.dev landing page.
- Confirm `dart pub publish --dry-run` reports no documentation warnings.

## [ ] ORT version upgrade pipeline

Define and document a repeatable process for adopting a new ONNX Runtime
version. The process must cover:

- Updating `VERSION_ONNX` and regenerating `lib/src/generated/versions.g.dart`
- Verifying that vtable slot indices in `lib/src/ort_api.dart` still match the
  new `onnxruntime_c_api.h`
- Updating `version_onnx.json` with real SHA-256 digests for all platforms
  (desktop, Android AAR + per-ABI `.so`, iOS SPM version) — `hook/build.dart`
  reads this file at build time; no inline manifest exists in the hook
- Checking whether any platform has a patch release that supersedes the baseline
  version (as Windows v1.22.1 did for v1.22.0) and updating the relevant
  `version_onnx.json` platform entry's `version` and `url` fields accordingly
- Confirming the ORT API version constant (`ortApiVersion`) is correct for the
  new release
- Running the full integration test matrix across all supported platforms before
  publishing

The primary deliverable is `tool/update_ort_version.dart` — a script that
accepts a target version tag, discovers the platform binaries from the GitHub
Releases page (handling platform-specific patch overrides), downloads each,
computes its SHA-256, and emits a ready-to-merge `version_onnx.json` update.
This eliminates the manual `curl | sha256sum` archaeology that produced the
v1.22.0 digests.

## Post-v0 backlog

These items are intentionally out of v0 scope. They are noted here to avoid
re-discovery and to inform future roadmap planning.

### ONNX external data format support

Some large models (e.g. full-precision BGE-M3) split weights into a separate
`.onnx_data` file. `createSession(modelBytes)` cannot load these models; the
external data path is untested with `createSessionFromFile`. The workaround for
v0 is to use self-contained single-file exports (quantized variants). Full
support would require testing `createSessionFromFile` with co-located external
data, documenting the constraint in the API, and potentially adding a
`SessionOptions.externalDataDir` parameter.

## [ ] pub.dev delivery

Publish both packages to pub.dev as the final gate for v0. The two packages must
be released in lockstep because `betto_onnxrt_ios` is tightly coupled to the ORT
version baked into `betto_onnxrt` via `VERSION_ONNX` and the hand-maintained
vtable slots.

Pre-publication checklist:

- All blockers in the Testing pipeline section are resolved.
- Real SHA-256 digests are in place for all platforms (no placeholder zeros).
- `betto_onnxrt` and `betto_onnxrt_ios` carry matching version numbers.
- `dart pub publish --dry-run` passes for both packages.
- `CHANGELOG.md` is up to date for both packages.
- Confirm `betto_onnxrt` scores well on [pub.dev](https://pub.dev) package
  analysis (documentation, platform coverage, static analysis).

Publication order:

1. Publish `betto_onnxrt` first (no Flutter dependency; pure Dart).
2. Publish `betto_onnxrt_ios` immediately after (depends on Flutter SDK only,
   not on a pub.dev version of `betto_onnxrt`).
